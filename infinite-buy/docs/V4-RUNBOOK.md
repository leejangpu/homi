# 무한매수법 V4.0 런북 (시작 방법 / 전환 절차)

## 1. V4.0이 기존(v2.2/v3.0)과 다른 점 (요약)
라오어 **V4.0**은 카페 공식 + 언이시트(회원 '언이'의 자동계산 Apps Script)를 1:1 포팅한 것. 상세는 [`../V4-SPEC.md`](../V4-SPEC.md).

| 항목 | v2.2/v3.0 | **v4.0** |
|---|---|---|
| 1회매수금 | 원금/분할수 (고정) | **잔금/(분할수−T)** (동적, 후반 백로딩) |
| T값 | 누적투자금/1금 (재계산) | **체결 이벤트 누적** (전량+1/절반+0.5/쿼터매도×0.75, 경로의존) |
| 매수 주문 | 별%/평단 LOC 1~2건 | 별지점/평단 LOC + **큰수(종가×1.10)** + **대폭락 5단 티어** |
| 소진 후 | v2.2 쿼터모드 / v3.0 잔금부족 손절 | **리버스모드**(5일평균 별지점, MOC 등분매도, 쿼터매수, T 감쇠) |
| 별% 공식 | 동일 | 동일 (target−2·target/n·T) |
| 매도 | 1/4 별 LOC + 3/4 목표 LIMIT | 동일 |
| 원금 | principalCalculator | **실계좌 예수금 배분 → 이후 복리** |

> 백테스트 비교(원금 $10k, 40분할): 강세장에서 v4가 우위(TQQQ +56% vs v3 +44%, SOXL +117% vs v3 +94%), 깊은 폭락장은 v2.2가 방어 우위. 자세한 수치는 `compare.ts`로 재현.

## 2. 현재 상태 (코드 준비 완료, 안전)
- 구현·검증 완료: 코어 앵커 17/17, 포팅 앵커 19/19, 결선↔백테스트 5/5(리버스 포함), 실주문 0건.
- **`config-v4.json` `enabled:false`** → 워크플로 돌려도 아무 동작 안 함.
- 기존 v2.2 실거래 경로 무수정.

## 3. 시작 절차 (단계적, 각 단계 검증 후 다음)

### STEP 0 — 전제
- `infinite-buy/.env`에 KIS 키(기존 v2.2와 동일 계좌)·텔레그램 설정 존재.
- v4 종목/분할/목표 확인: `config-v4.json` (기본 TQQQ 40분할 목표15%, SOXL 40분할 목표20%, 큰수10%). 필요 시 분할수 등 조정.

### STEP 1 — 기존 사이클 자동재시작 정지
- `config.json`의 `"autoRestart": true` → **`false`**.
- 효과: 현재 진행 중인 v2.2 사이클이 **종료되면 새 사이클을 시작하지 않음**. 종목이 비워짐(보유 0).

### STEP 2 — V4 DRY-RUN 검증 (주문 없음)
- `config-v4.json` `"enabled": true`.
- v4-close 워크플로(또는 `npx tsx src/v4/main-close-v4.ts`) 1회 실행.
  - 전환 가드: v2.2가 아직 보유 중인 종목은 "v4 대기" 로그 → 정상.
  - v2.2가 비워진 종목은 v4 활성화(첫 사이클 원금=예수금 배분) + `state/v4-<종목>.json`·로그 생성.
- 로그에서 sma5/예수금/주문계산이 정상인지 눈으로 확인. **이 단계까지 실주문 0건.**

### STEP 3 — 실주문 ON
- v4-open 워크플로를 입력 `live_orders: YES_REALLY`로 실행(또는 env `V4_LIVE_ORDERS=YES_REALLY`).
- 이게 있어야만 `submitOrder`가 실제 KIS로 나감. 없으면 영원히 DRY-RUN.

### STEP 4 — 스케줄 자동화
- `infinite-buy-v4-{open,close}.yml`의 `schedule:` 주석 해제(또는 launchd plist 추가). 타이밍은 기존 v2.2와 동일 개념(개시 전 open, 마감 후 close).

## 4. 전환(인수) 메커니즘 — 종목별 자동
`runCloseV4`가 종목마다:
- 기존 `state/<종목>.json`이 **active이거나 KIS 보유>0** → v4 대기(건드리지 않음).
- v2.2가 완전히 빠진(보유 0·완료) 종목만 → v4가 인수, 첫 사이클 시작.

따라서 TQQQ·SOXL이 **각자 끝나는 시점에 따로** v4로 넘어간다. 둘 다 한 번에 갈아탈 필요 없음.

## 5. 안전 가드 (실주문 3중)
1. `config-v4.json` `enabled:false` → 즉시 종료
2. env `V4_LIVE_ORDERS=YES_REALLY` 없으면 모든 주문 DRY-RUN(로그만)
3. DRY-RUN 시 `kis.submitOrder` 자체 미호출

## 6. 롤백
- v4 멈춤: `config-v4.json` `enabled:false` (또는 `V4_LIVE_ORDERS` 제거 → DRY-RUN).
- v2.2 복귀: `config.json` `autoRestart:true` 되돌리면 기존 전략이 다시 사이클 시작.
- v4 상태는 `state/v4-*.json`에만 있어 기존 상태와 독립.

## 7. 미구현(선택) — 실가동 시 추가 고려
- v4 텔레그램 알림(현재 콘솔/로그만).
- 모의투자(VTS) 서버 라우팅(현재 실전 도메인만).
